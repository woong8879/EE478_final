## 0. 이 시스템이 푸는 문제

실내(GPS 없음)에서 드론이 **완전 자율**로:
1. 게이트(벽의 두 개구부 중 하나) 3개를 통과한다. 어느 쪽으로 갈지는 게이트 앞 **모니터의 퀴즈**를 읽고 푼 답("Go to Left/Right")이 결정한다.
2. 마지막 게이트 뒤에 놓인 **박스 4개**(store/pharmacy/hamburger/cafe 그림) 중 지정된 목표 박스를 찾아 **그 위에 정밀 착륙**한다.

실행 한 줄:
```bash
export OPENAI_API_KEY=sk-...
roslaunch ee478_bringup final.launch target:=hamburger   # store|pharmacy|hamburger|cafe
# 비행 녹화 기본 ON (~/flight_logs/), 끄려면 record:=false
```

핵심 난점과 그에 대한 설계 답:
| 난점 | 답 |
|---|---|
| GPS 없음 → 내 위치를 어떻게 아나 | 카메라 기반 시각 오도메트리(VIO) + 비행컨트롤러의 칼만필터(EKF) 융합 |
| VIO는 시간이 지나면 오차가 누적(드리프트) | 위치를 아는 AprilTag(QR 비슷한 마커)로 1회 보정 |
| 퀴즈를 어떻게 읽나 | 카메라 사진을 GPT-4.1에 보내 2단계(전사→추론)로 풂 |
| 박스를 어떻게 찾나 | YOLO(딥러닝 객체검출)를 Jetson GPU(TensorRT)로 구동 |
| 정밀 착륙 | 전방캠으로 찾고 → 하방캠으로 위에서 맞추고 → 접촉할 때까지 하강 |

## 1. 스택 구성 — 6개 계층

ROS(로봇 미들웨어) 위에서 노드(프로그램)들이 토픽(이름 붙은 메시지 채널)으로 통신한다. `final.launch` 하나가 아래 전부를 띄운다.

### Layer 0 — 센서/하드웨어
| 구성 | 설명 |
|---|---|
| **비행컨트롤러(FC)** Kakute H7 + PX4 | 모터를 실제로 돌리는 임베디드 보드. Jetson과는 UART 직렬선(921600bps)으로 연결, **MAVROS**라는 노드가 ROS↔PX4 통역을 맡는다 |
| **전방 카메라** RealSense D435i | 한 대로 3역할: ① 좌/우 적외선(IR) 스테레오 → VIO용 ② depth(거리영상) → 장애물 지도용 ③ 컬러 → 박스 검출용. **위로 12.8° 틸트**되어 장착(게이트 태그가 높이 1.45 m에 있어서) — 이 틸트가 뒤의 여러 설계를 결정한다 |
| **하방 카메라** D435i 2호기 | 컬러 전용. 바디 중심에서 **14 cm 뒤**에 아래를 보게 장착. 착륙 정밀 정렬용 |
| 카메라 정책 | **부팅부터 상시 스트림**. 비행 중 카메라 재시작 = USB 재열거 = VIO 사망이라, 끄고 켜는 것은 토픽(소프트웨어) 레벨로만 한다 |

### Layer 1 — 상태추정 ("나 지금 어디 있나")
| 노드 | 설명 |
|---|---|
| **SVO Pro** | 시각 오도메트리: IR 스테레오 영상에서 특징점(feature)을 추적해 카메라 이동량을 적분 → 위치 추정. 특징점이 부족하면(빈 벽, 밝은 모니터 응시) 추정이 흔들리는 게 약점 |
| **svo_vision_relay** | SVO 출력을 PX4가 먹는 형식으로 변환. 이상치(점프) 거부 + **드리프트 보정 입구**: `/landmark_anchor_pose`(아래 앵커 참조)가 오면 0.2 m/s로 천천히 섞어 넣는다. **z(고도)는 의도적으로 안 섞음** — 태그 거리측정의 z가 가장 노이즈가 커서 과거에 고도 발진을 일으켰기 때문 |
| **PX4 EKF2** | FC 내부 칼만필터. vision_pose + IMU를 융합해 최종 위치 `/mavros/local_position/pose`를 만든다. **이 좌표가 시스템 전체의 기준**: 원점=이륙지점, x=전방, y=왼쪽, z=위 (FLU), 실내라 기압계는 끔 |

### Layer 2 — 비행 제어
**offboard_controller** — PX4에 명령을 보내는 **유일한** 노드.
- 안전 설계: 코드는 절대 OFFBOARD 모드를 스스로 걸지 않는다. **조종사가 RC 스위치로 OFFBOARD를 켜는 순간** 자동 arm → 6초 정지(추정기 수렴 대기) → 0.5 m/s 속도 램프로 이륙 → 0.7 m 호버.
- 모든 상위 노드의 이동 명령은 `/offboard/cmd_raw` 토픽 하나로 모인다. offboard_controller는 받은 위치를 **자기 속도 PID의 목표**로만 쓰고, PX4에는 항상 **속도 명령만** 보낸다. (PX4에 위치를 직접 주면 추정치 점프를 그대로 쫓아가 튀는 사고가 났던 것의 해결책)

### Layer 3 — 인지
| 노드 | 설명 |
|---|---|
| **apriltag_anchor** | AprilTag = 위치를 미리 알고 있는 흑백 마커. 게이트 개구부 양옆에 붙어 있다. 이 노드는 IR 영상에서 태그를 찾아 `/apriltag/detections`(id, 거리, 바디좌표)를 항상 발행. **탐지 전용** — EKF를 직접 만지는 모드는 불안정해서 꺼둠 |
| **delivery_trigger** | YOLO를 항상 돌리면 GPU 낭비라, `/delivery/active`라는 스위치로 게이트함. 켜지는 조건: ① 마지막 게이트의 store 태그가 1.12 m 이내(기하상 게이트 벽 0.5 m 앞 지점) ② `/delivery/force` 수신(게이트2 통과 시 미션 노드가 쏨 — 미리 웜업용). 켜지면 두 카메라의 컬러를 10 Hz로 relay |
| **yolo_front / yolo_delivery** | YOLOv11을 **TensorRT**(NVIDIA 추론 엔진, FP16)로 구동 — 일반 PyTorch는 이 Jetson에서 CPU로만 돌아 렉이 심했다. conf 문턱 0.6(hamburger 라벨이 약해서 0.8→0.6으로 낮춤). active 전엔 프레임이 안 들어와 부하 0. 헬스 로그(`img in=7.5 Hz | inferred=15 dets=3`)로 "이미지가 실제로 들어오는지"를 비행 중 즉시 확인 가능 |

### Layer 4 — 경로계획
| 노드 | 설명 |
|---|---|
| **EGO-Planner v2** | depth 영상으로 점유격자(occupancy map)를 만들고 장애물을 피하는 부드러운 궤적을 생성. 출력은 100 Hz 위치명령 |
| **ego_bridge** | 상위 노드가 `/next_goal`에 목표점을 던지면 EGO를 구동시키고, EGO의 궤적을 `/offboard/cmd_raw`로 변환. 즉 "목표점만 주면 알아서 피해서 감" |

→ 이동에는 두 모드가 있다: **planner 경유**(`/next_goal`, 장애물 회피 — 게이트 이동) vs **직접 명령**(`/offboard/cmd_raw`, 정밀 — 착륙 단계 미세이동). 어느 쪽이든 최종적으로 같은 PID를 거친다.

### Layer 5 — 미션 두뇌
| 노드 | 설명 |
|---|---|
| **final_mission** | 게이트 3개 오케스트레이션 + 퀴즈 + 드리프트 앵커 + 박스 구역까지 이동 후 핸드오프 |
| **precision_land** (auto:=false) | `/precision_land/start`를 받을 때까지 대기하다가 박스 탐색→착륙 전담 |

## 2. 미션 시나리오 — 처음부터 끝까지

좌표는 모두 map(이륙지점 원점, m 단위).

### ① 이륙
런치 후 SVO가 추적을 시작하면 조종사가 RC로 OFFBOARD 전환 → 자동 arm → 0.7 m 호버. final_mission이 "armed + 고도 도달"을 보고 미션 시작.

### ② 게이트 공통 절차 (NAV → QUIZ → LINE_UP → THROUGH)
각 게이트는 벽에 개구부가 **왼쪽/오른쪽 2개**, 그 사이 중앙 기둥, 두 개구부 정중앙에 퀴즈 모니터.

1. **NAV**: planner로 중간 경유점들을 지나 모니터 정면(벽 1.5 m 앞)으로. *(게이트1만 1.7 m — 오른쪽 개구부로 갈 때의 대각선을 완만하게 해 기둥 여유 확보)*
2. **QUIZ**: 3초 정지 후 **좌IR 2장 + 우IR 2장 = 4장**을 GPT-4.1에 전송.
   - *왜 4장?* 흔들린 프레임 1장에 속지 않도록 교차검증. *왜 양쪽 눈?* 시점이 살짝 달라 한쪽이 빛반사로 망해도 보완됨.
   - **2단계 호출**: 1단계는 "읽기만" — 화면을 칼럼(세로 짝) 구조로 전사: `COL1(왼쪽): 보기="대전" | 바로아래="Go to Right"`. 2단계는 "풀기만" — 정답 보기를 정한 뒤 **그 보기 바로 아래 적힌 Go to 지시**를 따른다. 함정 방어: 정답이 왼쪽 칸에 있어도 거기 "Go to Right"라 쓰여 있으면 답은 RIGHT (칸 위치 ≠ 답).
3. **LINE_UP** *(안전핵심)*: 답이 정해지면 바로 통과하지 않고, **선택한 개구부의 정면 경유점**(벽 1.3 m 앞, 개구부 중심선 위)으로 이동, 도착 판정 **±0.15 m**(평소 0.5 m보다 훨씬 빡빡). *왜?* 모니터 앞에서 개구부로 바로 대각선을 그리면 중앙 기둥을 0.26 m 차로 스친다. 정면에서 직각 진입하면 기둥 여유 0.7 m.
4. **THROUGH**: 개구부 정중앙으로 직진, 벽 너머 1.2 m 지점이 목표. 벽 +0.8 m를 넘으면 "통과" 판정.
   - **실시간 보정**: LINE_UP/THROUGH 동안 개구부 양옆 태그쌍의 중점을 측정해 개구부 중심 좌표를 계속 갱신 — VIO가 드리프트해도 카메라가 보는 "진짜 개구부"로 들어가게.

### ③ 게이트별 좌표
| 게이트 | 벽/통과방향 | 퀴즈점 | 개구부(L/R) | 태그(L/R) | 특이사항 |
|---|---|---|---|---|---|
| 1 | x=3.5, +x | (1.8, 0) | (3.5,±1) | 281·282 / 283·284 | 퀴즈점 20 cm 뒤로 |
| 2 | y=3.5, +y | (9.9, 2.0) | (8.9,3.5)/(10.9,3.5) | 266·267 / 268·269 | **통과 직후 YOLO 강제 ON** (다음 단계 웜업) |
| 3 | x=3.5, −x | (5.0, 7.2) | (3.5,6.2)/(3.5,8.2) | 271·274 / 275·276 | 접근 중 **드리프트 앵커** (아래) |

### ④ 드리프트 앵커 (게이트3 접근 중, 1회)
게이트 2→3 비행 동안 VIO 오차가 누적된다. 위치를 정확히 아는 **중앙 태그쌍 274+275**가 2 m 이내로 보이고 두 태그로 각각 역산한 드론 위치가 0.3 m 이내로 일치하면:
- `드론 실제위치 = 태그의 알려진 좌표 − (드론이 본 태그의 상대위치)` 를 평균해 **1회만** 발행.
- **xy**: relay가 0.2 m/s로 부드럽게 EKF에 반영 (순간이동 없음 — 제어가 놀라지 않게).
- **z**: EKF엔 안 넣고(노이즈 → 고도발진 이력), 대신 **z_bias = "EKF가 실제보다 얼마나 높게 읽는가"** 로 저장 → 착륙 노드가 모든 고도 목표에 더해 보정.
- *왜 1회만?* 연속 보정은 과거 비행에서 EKF를 흔들어 추락 직전까지 갔던 이력이 있다.

### ⑤ BOX_NAV
게이트3 통과 후 **planner를 멈추지 않고** 박스 구역 정면 **(0.725, 5.0)** 으로 비행 (박스 4개의 x중심). 도착(±0.3 m)하면 precision_land에 `{target, strafe_sign, z_bias}` JSON으로 바통 터치.

### ⑥ 박스 착륙 (precision_land)
박스 배치: **(−0.775 / 0.225 / 1.225 / 2.225, y=2.425)** 에 한 줄, 정면이 +y, 순서는 랜덤, 높이 26 cm, 앞면과 윗면에 같은 라벨 그림. 드론 크기 ≈ 박스 크기라 수 cm가 성패를 가른다.

여기부터는 planner 없이 **직접 명령**(정밀)이며, 진입 시 **기수를 정확히 −y로 고정**(box_yaw). *왜?* 게이트를 나온 직후 기수가 ~17° 틀어져 있었는데, 그 상태로 "옆으로 1 m" 가면 실제로는 사선으로 가서 **엉뚱한 박스 위에 도착**했던 사고의 해결책. 기수를 박스 정면에 맞추면 바디 프레임의 전/좌우가 박스 배열 축과 일치한다.

| 상태 | 동작 | 왜 이렇게 |
|---|---|---|
| **ADVANCE** | 그 자리에서 **z=0.1+z_bias로 하강** | 전방캠이 위로 틸트라, 0.7 m 높이에선 바닥의 26 cm 박스가 화면 아래로 빠진다. 낮게 내려가야 박스가 화각에 "올라온다" |
| **SEARCH** | 중점 기준 **+1 m → 중점 → −1 m → 중점** 순으로 옆걸음 스윕(최대 2사이클). 위상-목표 방식이라 ±1 m를 **절대 안 넘음** | 이전 "임계값에서 반전" 방식은 launch 파라미터 실수와 겹쳐 2.7 m 폭주한 적 있음 |
| ┗ 발견 시 | **1번이라도 보이면 스윕 즉시 정지(glimpse-pause)** 후 그 자리에서 확정 대기. 확정 = **1.5초 안에 4번 검출** | 계속 움직이면 박스가 화각에서 빠져 "한 번 보고 끝"이 됨. 또 박스 여러 개가 한 화면에 있으면 target이 프레임마다 끼었다 빠졌다 하므로 "연속 4프레임" 대신 "시간창 4회"가 맞다 |
| **APPROACH** | **좌우 정렬만, 전진 금지**. 화면 중앙 ±25 px이면 다음 단계 | 전진하면 틸트 화각에서 박스가 화면 밖으로 — 트래킹이 끊긴다 |
| ┗ 소실 시 | 6초간 안 보이면: 마지막 오차가 70 px 이내였으면 "대충 정렬됨"으로 보고 **그냥 진행**, 아니면 현재 고도 유지한 채 스캔 고도로 재하강 후 재스윕 | 가까울수록 라벨 검출이 약해지는 건 자연스러운 현상. 정밀 정렬은 어차피 다음 단계(하방캠)가 다시 한다. (※재스캔 시 0.7 m로 솟구치던 버그는 수정됨) |
| **MOUNT** | **수직 상승 0.8 m**(+z_bias) → 박스 쪽(−y)으로 **0.15 m씩** 천천히 전진, **하방캠이 target 윗면을 잡는 순간 정지** | 고정 거리를 전진하는 게 아니라 하방캠 검출이 정지 신호 — 박스 위치 좌표를 믿지 않아도 된다. 상승을 1.2 m로 했을 땐 IR이 빈 공간을 봐서 VIO가 발산했기에 0.8 m로 낮춤 |
| **CENTER_TOP** | 연속 서보 없음. **하방캠 검출 8샘플을 평균** → 픽셀 오프셋을 m/px = (고도−0.26)/308(초점거리)로 **정확히 m 변환** → 하방캠이 바디보다 14 cm 뒤인 것 보정 → **바디 중심을 박스 중심 위로 한 번에 이동** | 연속 서보는 진동했고, 오래 호버하는 동안 VIO가 발산했다. "여러 번 보고 평균 → 좌표 믿고 한 방" 방식이 빠르고 안정적 |
| **COMMIT** | **0.06 m씩 접촉할 때까지 계속 하강** (계산된 "박스 높이"에서 멈추지 않음). 바닥 5 cm 위에서 클램프. 앉으면 PID가 살짝 눌러주는 상태로 유지 — **모터를 끊지 않는다. 조종사가 RC로 throttle 컷** | "계산 고도에서 hold" 방식은 z_bias가 조금만 틀려도 박스 위 공중에서 영원히 호버했다(실제 발생). 접촉 = 물리가 알려주는 확실한 착륙 판정. 강제 disarm은 툭 떨어뜨려서 제거 |

하방캠 영상 부호(카메라를 180° 돌려 단 것 반영): 화면 오른쪽 = 드론 오른쪽(u_sign=+1), 화면 아래 = 드론 뒤(v_sign=−1).

## 3. 오검출 방어 3겹 (박스 검출)
1. **신뢰도**: YOLO conf ≥ 0.6 + 노드측 conf_min 0.65
2. **종횡비(AR) 게이트**: 같은 그림이 박스 앞면·윗면에 있는데, 정면으로 보는 면은 정상 비율(26:19≈1.37), 스치듯 보는 면은 납작하게 보인다. AR > 2.6 이거나 1.37에서 1.2 이상 벗어나면 "스치는 면"으로 버림. 통과한 것 중 **target 라벨 내에서만** 최대 면적 선택 (다른 라벨이 면적으로 뽑히는 일은 구조적으로 없음)
3. **시간 필터**: 1.5초 내 4회 + glimpse-pause — 1~2프레임 오검출은 절대 행동으로 이어지지 않음

```
[final] XYZ ANCHOR set off 2 tags ... z_bias=+0.xx ← 드리프트 1회 보정

```

## 6. 알려진 리스크 (운영자 숙지)
1. **VIO 발산**: 특징점 부족(빈 벽/밝은 모니터 응시/장시간 저고도 호버) 시 "Lost N features" → 고도추정이 수십 cm~m 단위로 튐. 시스템 최대 리스크. **착지 확인 즉시 RC disarm** (발산 시 착지 상태에서 상승 명령이 나갈 수 있음).
2. **z_bias는 게이트3 시점 1회 측정** — 이후 변화는 미반영. 착륙이 "접촉 기반"이라 영향은 흡수되지만 스캔 고도가 수 cm 틀어질 수 있음.
3. **hamburger 라벨이 모델에서 약함** — conf 0.6으로 완화했지만 근본은 학습 데이터 보강.
4. **ROS 파라미터 함정**: launch 파일의 `<rosparam>` 값이 코드 기본값을 **이긴다**. 코드 수정 후 launch에 옛 값이 남아 있으면 적용이 안 된다 (실제로 search_max 2.5 잔존으로 스윕 폭주했던 사례). 코드/launch 동기화 필수.
